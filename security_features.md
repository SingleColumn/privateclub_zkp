# Security Architecture Overview — Proof of Funds Credential App

**Author:** Lead Developer  
**Date:** 2026-02-26  

---

## 1. Purpose

This memo outlines the security controls implemented in the Proof of Funds Credential App. The goal of the system is to make it difficult for members to falsely claim available funds while preserving privacy and minimizing operational complexity.

The app is designed to deter casual and moderately sophisticated tampering. It is not intended to provide bank-grade guarantees or replace escrow/custody mechanisms.

---

## 2. Security Design Principles

The system follows these principles:

1. **Member privacy by design** — No exact balances are shared. Only tier-level claims are revealed.
2. **Client integrity enforcement** — Credentials are accepted only if generated by the official app running in a non-compromised environment.
3. **Cryptographic binding** — Proofs are cryptographically bound to credential payloads.
4. **Key isolation** — Sensitive keys never leave secure hardware storage.
5. **No reliance on obscurity** — Security does not depend on secret client logic.

---

## 3. Device & App Integrity Attestation

To reduce the risk of tampering, the app uses platform-backed integrity attestation.

### iOS

- **Apple App Attest**
  - App generates a hardware-backed key in Secure Enclave.
  - Apple signs an attestation object confirming:
    - App is the official bundle ID.
    - App binary is signed by the registered developer.
    - Key was generated in secure hardware.
  - Verifier validates Apple’s signature and attestation payload.

### Android

- **Google Play Integrity API**
  - App requests integrity token from Google.
  - Token includes:
    - App integrity (official Play-signed build).
    - Device integrity (non-rooted / non-emulated).
  - Token signature validated before accepting credential.

### Policy

- Credentials generated without valid integrity attestation are either:
  - Rejected, or
  - Marked as “Untrusted environment” (configurable policy).

Integrity attestation significantly reduces risk of:
- Repackaged apps
- Modified binaries
- Basic rooted/jailbroken environments
- Emulator-based attacks

---

## 4. Cryptographic Key Security

### Key Storage

- Private keys are generated and stored in:
  - **Secure Enclave (iOS)**
  - **Android Hardware-backed Keystore**
- Keys are:
  - Non-exportable
  - Bound to the device
  - Protected by OS-level isolation

### Key Usage

- Keys are used to:
  - Bind credential payloads to a device
  - Sign proof data
- Keys never leave secure hardware.
- No private keys are stored in plaintext or transmitted externally.

---

## 5. Proof Generation Model (Current Stage)

In the current prototype stage:

- ZKP functions are **API-structured placeholders**, not real cryptographic zero-knowledge proofs.
- Proof generation and verification are simulated using deterministic hashing logic.
- This enforces architectural separation between:
  - Member-side proving
  - Verifier-side validation

Future production versions may replace stub logic with real range proofs or SNARK-based constructions.

---

## 6. Data Minimization

The app does not:

- Share exact balances.
- Share account numbers.
- Share bank names.
- Transmit financial data to club servers.

Only the following are included in credentials:

- Tier ID
- Timestamp
- Integrity status
- Pseudonymous member identifier

---

## 7. Anti-Tampering Controls

In addition to platform attestation:

- Jailbreak/root detection heuristics.
- Debugger detection.
- Emulator detection.
- Obfuscation of sensitive code paths.
- Runtime integrity checks where applicable.

These controls increase the difficulty of runtime manipulation but are not relied upon as sole trust anchors.

---

## 8. Token Security

- Credentials are encoded as opaque tokens (e.g., base64url JSON).
- Tokens do not contain readable financial information.
- Credential payload and proof are cryptographically bound together.
- Malformed or tampered tokens are rejected during verification.

---

## 9. Known Security Limitations

The system does **not** protect against:

- Highly skilled reverse engineering on jailbroken/rooted devices.
- Runtime hooking by sophisticated attackers.
- Temporary balance manipulation (“borrowed funds”).
- Credential forwarding to third parties.

The system is designed to:
- Stop casual fraud.
- Deter moderate technical abuse.
- Raise the cost of deception.
- Provide visible assurance in a social context.

---

## 10. Open Source Considerations

If open sourced:

- Security remains anchored in:
  - Platform signatures
  - Hardware-backed keys
  - Apple/Google attestation signatures
- Security does not rely on code secrecy.
- Modified builds will fail integrity validation.

---

## 11. Summary

This app implements layered security:

1. Platform-backed integrity attestation  
2. Hardware-protected private keys  
3. Cryptographic binding of credential payloads  
4. Strict data minimization  
5. Client-side tamper detection  

While not bank-grade custody infrastructure, the system provides a strong deterrent against casual and mid-level tampering, aligned with the club’s objective of filtering posers and reducing social friction.
